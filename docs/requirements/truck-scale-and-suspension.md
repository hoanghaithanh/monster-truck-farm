# Truck Scale & Independent Wheel Suspension

Status: Draft — three decisions are locked in by direct human confirmation (see below); remaining items are non-blocking open questions for the architect/developer before this is sized and scheduled.

Related: `docs/requirements/truck-builder-and-upgrades.md` (the 3 body tiers and their existing size progression this doc scales up); `docs/requirements/truck-obstacle-climbing.md` (the existing whole-body lift/tilt effect and its four-corner-sampling mechanism this doc's suspension layers on top of, per ADR 0013/0014); `docs/requirements/truck-wheel-motion.md` (the existing per-wheel roll/steer-yaw animation this doc's suspension must coexist with, not replace); `docs/requirements/vehicle-and-character-art.md` (the tier-to-tier visual-progression design intent this doc must preserve); `docs/requirements/drive-terrain-and-gas.md` (AC5-AC9, the obstacle-clearance rule this doc does not change); `docs/requirements/animal-chase-and-coins.md`, `docs/requirements/farmer-minimal-bump.md`, `docs/requirements/fuel-pickups.md` (spawn/contact systems whose keep-out distances are downstream of the truck's contact radius and are in scope for re-tuning here); ~~the sibling hills/terrain requirements pass (in progress in parallel as of this writing, not yet published as a doc)~~ **update (2026-07-11): published and shipped — `docs/requirements/terrain-expansion-and-hills.md` / `docs/architecture/0017-terrain-expansion-and-visual-hills.md`, issue #49, signed off (see `docs/acceptance/sprint-5-issue49-terrain-hills-2026-07-11.md`)** — this doc's suspension mechanism must not structurally preclude that epic's later hill-following behavior, but does not depend on it landing first.

## Problem statement

Direct player feedback (roadmap ask #4, this session): "Make the truck bigger, add suspension effect when it climbs on object." Today the truck's on-screen size is tuned independently of its collision/contact size (`TRUCK_CONTACT_RADIUS`, `src/core/driving/config.ts`, a single flat constant reused by the physics collider and every gameplay contact check), and the existing obstacle-climb response (`docs/architecture/0014-obstacle-climb-four-corner-sampling.md`) moves the whole truck body as one rigid unit rather than articulating each wheel independently. The player wants a truck that reads as bigger/more imposing, with a collision size that actually matches what's on screen, and wheels that visibly move independently over bumps like a real vehicle's suspension — not just a scaled-up version of the current whole-body tilt.

Two decisions were confirmed directly by the human during this session's planning pass and are treated as settled requirements, not open questions:

1. **The collision hitbox scales proportionally with the bigger visual size.** Chosen over "visual only, hitbox unchanged." This means every distance in the game tuned against today's hitbox (obstacle-climb footprint, animal/farmer/fuel spawn keep-out, obstacle clearance feel) needs to be re-checked and, where needed, re-tuned alongside it.
2. **Suspension is true independent per-wheel vertical travel.** Chosen over "enhance the existing whole-body tilt effect." Each of the 4 wheels must visibly move up/down on its own; the existing whole-body tilt stays and layers with this, it is not replaced.
3. **The size increase applies to all 3 body tiers proportionally, preserving today's tier-to-tier relative size progression.** Chosen over "only the top tier grows." Confirmed 2026-07-10 — resolves what was Open Question 1 below; AC1/AC2 already assumed this as the recommended default and are now settled requirements, not provisional.

## Goals / Non-goals

**Goals**
- Increase the truck's visual size across all 3 body tiers (see Open Question 1 for the recommended default on *how* the scale-up applies across tiers) so the truck reads as bigger/more imposing to a player at a glance.
- Preserve the existing, already-meaningful tier-to-tier size progression (Tier 0 < Tier 1 < Tier 2, per `truck-builder-and-upgrades.md`'s body-tier data and `vehicle-and-character-art.md`'s design rationale for why visible tier progression matters to a young player) — the scale-up must not flatten or erase the relative size difference between tiers.
- Scale the truck's collision/contact radius proportionally with the new visual size, using the same scale factor applied to the model, for every tier (per the human's locked-in decision above).
- Re-check and re-tune every downstream distance that was tuned against today's smaller hitbox — obstacle-climb footprint math, animal/farmer/fuel-pickup spawn keep-out distances, and obstacle-clearance "feel" — so the bigger truck doesn't make the game feel newly cramped, newly unfair, or (in the other direction) trivially easy as an unplanned side effect.
- Give each of the 4 wheels true independent vertical suspension travel that visibly reacts when that specific wheel's corner is over an obstacle, distinct from the other three wheels' current position — not a single shared value applied to all four.
- Layer this per-wheel motion on top of the existing whole-body lift/tilt effect (`truck-obstacle-climbing.md`) — both effects visible together, neither replacing the other.
- Suspension must be observable and testable today, over the existing discrete bush/rock/derelict-car obstacle instances, across all 3 wheel tiers — independent of whether the sibling hills/terrain epic has landed yet.
- Existing wheel roll (rotate on axle, speed-driven) and front-wheel steer-yaw (`truck-wheel-motion.md`) must keep working correctly alongside the new suspension vertical offset.

**Non-goals**
- Any real rigid-body/vertical-physics change to the truck's Rapier collider or the kinematic character controller — the truck's actual movement/collision resolution stays exactly as specified in `drive-terrain-and-gas.md` and ADR 0001 §2. Suspension here is a visual approximation layer, the same category of decision already made for the existing whole-body tilt effect.
- Hill-terrain-following suspension behavior — once the sibling hills/terrain epic lands, wheels reacting to rolling hill terrain (not just discrete obstacles) is that epic's concern to specify and gate. This doc only requires that the suspension *mechanism* not be built in a way that structurally rules that out later.
- Any change to the actual gameplay meaning of the four upgrade axes — hit capacity (body), obstacle-clearance class (wheels), top speed (engine), gas capacity (gas tank) are unchanged in value for every tier. This is a visual-feel and hitbox-consistency change, not a rebalance.
- Any change to which obstacles are `blocking` vs. `passable`, or the wheel-tier/obstacle-class clearance rule itself (`core/clearance.ts`'s `canClear()`) — untouched by this doc.
- New animation categories beyond suspension travel — tire deformation, dust/particle effects, drift behavior, skid marks (already a non-goal in `truck-wheel-motion.md`, carried forward here).
- Animating the builder screen's stationary 3D truck preview with suspension motion — there is no driving/obstacle state to animate from there (same precedent as `truck-wheel-motion.md`'s non-goal for that same preview). The preview should still reflect the new bigger size, just not suspension motion.
- Any change to animal/farmer AI behavior or spawn *logic* beyond the keep-out *distance* values needed to stay consistent with the bigger hitbox — this doc does not touch chase/flee/wander behavior.

## User stories

1. As a player, I want my truck to look and feel bigger and more imposing, so it feels like a real monster truck.
2. As a player, I want the size difference between my body upgrade tiers to stay clearly noticeable after the truck gets bigger, so upgrading still feels meaningful.
3. As a player, I want the game to still feel fair and not overly cramped or too easy after my truck's hitbox gets bigger, so driving, chasing animals, and avoiding the farmer still feel balanced.
4. As a player, I want each wheel of my truck to move independently up and down when I drive over a bump or obstacle, so my truck looks like it's really driving over the terrain instead of moving as one rigid block.
5. As a player, I want my wheels to keep rolling and turning correctly while also showing this new up-and-down motion, so my truck doesn't look broken or glitchy.

## Acceptance criteria

- **AC1 (bigger overall silhouette, all tiers):** Given any of the 3 body tiers, when its rendered model is compared to today's shipped size for that same tier, then the truck's silhouette (length/width/height) is measurably and noticeably larger — large enough that a player recognizes "my truck got bigger" at a glance without needing to measure it. Exact magnitude/percentage is a tuning value, not fixed by this AC (see Open Question 2).
- **AC2 (tier-to-tier progression preserved):** Given the three body tiers rendered side by side after the scale-up, when a player compares them, then Tier 0 < Tier 1 < Tier 2 remains clearly and distinguishably ordered by size, at a relative size gap comparable to today's shipped progression — the scale-up must not flatten the tiers toward a uniform size or make two adjacent tiers hard to tell apart.
- **AC3 (hitbox scales with the model, same factor, every tier):** Given the truck's visual size increases per AC1, then the truck's collision/contact radius (today's `TRUCK_CONTACT_RADIUS` or its per-tier replacement) increases by the same proportional scale factor applied to that tier's visual model — not left at today's flat value while only the model grows, and not scaled by some other, undisclosed factor.
- **AC4 (downstream distances re-tuned, not just the truck):** Given the bigger hitbox from AC3, when the truck drives the existing terrain (obstacles, structures, animal/farmer/fuel-pickup spawn points), then the obstacle-climb footprint math, the animal/farmer/fuel-pickup spawn keep-out distances, and the obstacle-clearance feel have all been re-checked and, where needed, re-tuned so the game does not feel newly cramped, newly unfair, or unintentionally easier (e.g., contacting multiple animals/pickups in one pass that used to require separate passes) purely as a side effect of the bigger hitbox. Verified by human playtest against the pre-change baseline, not code inspection alone.
- **AC5 (functional stats unchanged, all tiers/axes):** Given a truck built at any body/wheel/engine/gas-tank tier, then that tier's hit capacity, obstacle-clearance class, top speed, and gas capacity values are numerically identical to their values before this feature — nothing in this doc changes what an upgrade axis *does*, only how the truck looks and what physical space it occupies.
- **AC6 (independent per-wheel articulation):** Given the truck driving over a `passable` obstacle (bush/rock/derelict car) it's rated to clear, when one wheel corner's footprint is over the obstacle while the other three are not, then that wheel visibly moves along a vertical travel path distinct from the other three wheels' current position at that same moment — the four wheels are not constrained to move together as a single rigid group.
- **AC7 (whole-body tilt effect still present, layered not replaced):** Given the same obstacle-crossing scenario already covered by `truck-obstacle-climbing.md` AC1, the existing whole-body lift/tilt response continues to occur exactly as it does today, with the new per-wheel motion visible on top of it simultaneously — one effect does not substitute for or suppress the other.
- **AC8 (works across tiers and obstacle classes):** The independent per-wheel suspension response applies consistently for every truck-tier/obstacle pairing already covered by `drive-terrain-and-gas.md` AC6-AC8 and `truck-obstacle-climbing.md` AC5 (e.g., Tier 0 over a bush; Tier 2 over a bush, rock, or derelict car) — not just one hand-picked case.
- **AC9 (roll/steer-yaw unaffected by suspension):** Given the truck is moving and/or steering while a wheel is also showing suspension travel over an obstacle, then that wheel's existing roll animation (`truck-wheel-motion.md` AC1-AC3) and, for front wheels, steer-yaw (AC4-AC6) continue to animate correctly at the same time as the vertical suspension offset — all applicable motions are visible together on that wheel, none is dropped or corrupted by the others.
- **AC10 (no chaotic motion):** The suspension motion never flips, spins, launches, or otherwise destabilizes the truck's read as a stable vehicle, and requires no precision timing/input from the player to trigger correctly — consistent with the young-child-friendly, forgiving design bias already established project-wide (ADR 0001 §1, mirrored in `truck-obstacle-climbing.md` AC3).
- **AC11 (no phantom motion on flat ground):** Given the truck driving on flat ground with no obstacle under any wheel, then no wheel shows vertical suspension offset — the effect activates only in response to something being under that specific wheel corner, not as a constant idle bounce/animation.
- **AC12 (no false blocking or damage):** The suspension visual response, like the existing whole-body tilt it layers with, never prevents or measurably slows the truck's forward progress over a `passable` obstacle, and never counts as a hit against body capacity or triggers a fail state — carries forward `truck-obstacle-climbing.md` AC2/AC4 unchanged.

## Open questions

1. ~~Which tiers get the size increase?~~ **Resolved 2026-07-10 — see locked-in decision #3 above.** All 3 tiers scale up proportionally.
2. **(Non-blocking, tuning) Exact magnitude of the size increase.** How much bigger (e.g., a specific percentage, or a target absolute dimension) is left to the architect/developer to propose and the human to confirm via playtest — any increase that satisfies AC1's "reads as bigger at a glance" bar is acceptable at requirements level. Same treatment as `truck-wheel-motion.md` Open Question 2's steer-angle tuning value.
3. **(Non-blocking, tuning) Suspension travel amplitude/timing.** How far each wheel visually rises/falls and how quickly (an analog to the existing `ClimbConfig.maxLift`/`tiltGain` knobs for the whole-body effect) is left to the architect/developer to define and tune against live playtest, same precedent as the whole-body tilt's own tuning constants.
4. **(Non-blocking, informational) Expected re-tuning ripple.** AC4 requires re-tuning obstacle-climb footprint math, spawn keep-out distances, and clearance feel — the exact new constant values aren't decided here and shouldn't be; this is called out so the architect sizes the story with that ripple in mind rather than treating it as a pure rig/geometry change scoped to the truck alone.

## Constraints

- Must not change the functional meaning of the four upgrade axes: hit capacity, obstacle-clearance class, top speed, and gas capacity stay numerically identical to today's values for every tier (AC5) — this is a visual-feel and hitbox-consistency change only, not a rebalance of the builder/upgrade system.
- Must not change the existing wheel-tier/obstacle-class clearance *rule* (`core/clearance.ts`'s `canClear()`/`partitionObstacles()`) — only the physical distances feeding into spawn/footprint math are in scope for re-tuning, not the rule itself.
- Whatever suspension mechanism is chosen must stay within the existing arcade-kinematic architecture (ADR 0001 §2) — no genuine vertical-physics axis on the collider/controller, consistent with the same constraint already accepted for the whole-body tilt effect in `truck-obstacle-climbing.md`.
- Must not fork the single-truck-rig-assembly invariant (ADR 0011 §4 — the driving scene and the builder preview both call the same `buildTruckRig()`) — the builder preview may reasonably omit suspension motion (no driving state to animate from, same precedent as `truck-wheel-motion.md`'s non-goal for that preview) but must still reflect the new bigger size.
- Every downstream consumer of the truck's contact/collision size — the physics collider, animal-boop contact, farmer-bump contact, fuel-pickup contact, and the obstacle-climb footprint sampling (today all reading `TRUCK_CONTACT_RADIUS` from `src/core/driving/config.ts`) — must be re-validated against the new bigger value(s), not just the one call site that happens to be touched first.
- Must not regress ADR 0001's target-player design bias (forgiving, predictable, never chaotic, never a fail state from this feature) — a hard constraint given the young-child audience, same standing constraint as `truck-obstacle-climbing.md`.
- Runs in-browser (Three.js + Vite + Rapier3D-compat), consistent with the rest of the render/physics layers — no new tech introduced by this doc.
