# Terrain Expansion & Visual Hills

Status: Draft — requirements pass, not yet architected. Soft-boundary visible-edge question (Open Question 3) resolved 2026-07-10 — deferred again, see below.

Related: `docs/requirements/drive-terrain-and-gas.md` (AC4, the existing soft-boundary mechanic this doc resizes but does not redesign, and Open Question 3, the still-unresolved "visible edge treatment" question — see Open Questions below); `docs/requirements/truck-obstacle-climbing.md` and `docs/architecture/0013-obstacle-climb-visual.md` / `0014-obstacle-climb-four-corner-sampling.md` (the existing stateless, per-frame, four-corner visual lift/tilt system this doc extends to sample real terrain height, not a new mechanism); `docs/requirements/environment-dressing.md` (the existing structures/river/mountain-landmark content that must remain intact — see AC3a there for the current single mountain landmark, which this doc does not duplicate or replace); `docs/architecture/0001-foundation-stack-and-structure.md` §2 (the kinematic-only, no-rigid-body-vehicle-dynamics decision this doc explicitly preserves); `src/core/terrain.ts` (current `TERRAIN_BOUNDS` and all hand-placed content this doc's bounds change affects).

Two sibling epics are being requirements-analyzed in parallel and are deliberately **not** covered by this doc (see Non-goals): a truck-size/suspension epic (shares the four-corner sampling code this doc also touches) and a farm-layout/fields epic (corn/wheat fields, re-placement of existing structures — needs this doc's final bounds decided first, so is sequenced after it, not bundled with it).

## Problem statement

The player's own stated feedback: "The ground is flat and unattractive, I want bigger map, with hill and slope, like a typical golf course." Today's drivable area is a flat 40×40 plane (`TERRAIN_BOUNDS` -20..20 on both axes) with zero elevation anywhere in the codebase — the ground is a literal flat plane in both physics and rendering, and every part of the truck's movement math is pure 2D. The "golf course" comparison implies gentle, rolling, rounded terrain spread across a much larger open area — not canyons, cliffs, or additional dramatic mountain peaks (the game already has one such landmark, the single collidable mountain from `environment-dressing.md` AC3a, which stays as the one dramatic feature). This doc makes the map materially bigger and gives its ground a rolling, non-flat visual character, while leaving the truck's actual movement, collision, and gameplay math untouched.

## Goals / Non-goals

**Goals**
- Expand the drivable terrain bounds to roughly 100×100, about 6x today's 40×40 area (confirmed by the human; see Constraints for the exact figure this doc assumes).
- Add a visual-only rolling-hills treatment across the expanded terrain so the ground reads as gently undulating, golf-course-like landscape rather than a flat plane, without turning any part of the map into a canyon, cliff, or second dramatic mountain feature.
- Extend the existing four-corner obstacle-climb visual sampling (`docs/architecture/0014-obstacle-climb-four-corner-sampling.md`) to also sample real terrain height and add it to the existing lift/tilt calculation, so driving over a hill produces the same kind of visible rise/tilt response the truck already gives when driving over a clearable obstacle — reusing that precedent, not inventing a new visual language.
- Keep the truck's actual movement, steering, speed, collision, obstacle clearance, spawn logic, and gas systems **completely unaffected** by hill placement — hills are purely an emergent rendering property layered on top of the existing flat kinematic model, exactly as the existing obstacle-climb visual is layered on top of flat kinematic movement today. This is a hard constraint, confirmed directly by the human, not an open design question — see `docs/architecture/0001-foundation-stack-and-structure.md` §2, which rejected real rigid-body vehicle dynamics specifically because an unpredictable/flippable truck was judged wrong for this game's young-child audience.
- Confirm the existing soft-boundary mechanic (`drive-terrain-and-gas.md` AC4, a position clamp against `TERRAIN_BOUNDS`) continues to behave sensibly at the new, larger bounds.
- Confirm all existing drivable content (the three obstacles, the four structures including the mountain landmark, the river) remains reachable and fully functional after the map grows around it.

**Non-goals**
- **Real physics/slope interaction of any kind.** The truck must never actually roll, slide, gain/lose traction, flip, or get stuck because of a hill's slope — hills have zero effect on the truck's real position/collision math. This is the core safety constraint of this doc (see AC8).
- **Corn/wheat fields or any other new farm-dressing content.** Tracked separately in the farm-layout epic.
- **Repositioning existing structures, obstacles, or the river.** This doc's recommendation (see Open Questions) is that existing content stays at its current hand-placed coordinates — a small, already-tested area within a much bigger, currently-empty map — until the separately tracked farm-layout epic redesigns placement holistically. Combining a bounds/hills change with a full re-layout in one doc would conflate two different kinds of risk (a numeric bounds change vs. a creative relayout) and block this doc's core "bigger map + hills" ask on a decision (where should everything go now that there's more room?) that doesn't need to be made yet.
- **Truck suspension or per-wheel physical simulation.** Tracked separately in the truck-size/suspension epic, even though it shares the same four-corner sampling code this doc also extends — the two epics change different inputs to that sampling (terrain height here, wheel/chassis geometry there), not the sampling's fundamental nature.
- **A new obstacle/collision class for hills.** Hills are never `blocking` or `passable` in the existing wheel-tier clearance sense (`core/clearance.ts`) — they aren't gated by wheel tier at all, because the truck's real position math doesn't interact with them.
- **Additional mountain landmarks, canyons, or cliffs.** The one existing mountain (`environment-dressing.md` AC3a) remains the map's single dramatic large-scale feature; hills are a smaller-scale, gentler background texture over the rest of the terrain, not a second tier of mountains.
- **Fairway-style mowed-grass texturing, sand traps, or other literal golf-course dressing.** The player's "like a golf course" comment is read here as a shorthand for "gentle rolling terrain, not flat," not a literal request for golf visual theming — see Open Question 2 if the human wants to treat any of that as a stretch goal.
- **Spawn rate/count rebalancing for animals, farmer, or fuel pickups.** Out of scope for this doc even though a 6x-bigger map may make today's tuned spawn frequency feel sparse — flagged as Open Question 4, not resolved here.

## User stories

1. As a player, I want a much bigger map to drive around on, so there's more room to explore.
2. As a player, I want to see hills and gentle slopes across the farm, so the world looks like a real, rolling landscape instead of a flat plane.
3. As a player, I want driving over a hill to visibly rise and tilt my truck the way driving over an obstacle already does, so the hills feel like real terrain, not just a painted-on bumpy texture.
4. As a player, I want my truck to never get stuck, flip, or behave unpredictably because of a hill, so driving stays simple and forgiving no matter where I go on the map.
5. As a player, I want the windmill, barn, farmhouse, mountain, river, and the bush/rock/derelict-car obstacles I already know to still be exactly where I left them and still work the same way after the map gets bigger, so nothing I already learned about the world breaks.

## Acceptance criteria

### Map bounds

- **AC1 (bounds expanded, ~6x area):** `TERRAIN_BOUNDS` is expanded from the current 40×40 (-20..20 both axes) to approximately 100×100 (roughly -50..50 both axes), an increase of roughly 6x the current playable area. The exact boundary numbers are an implementation detail for the architect (see Constraints); the requirement is the ~6x area increase, confirmed by the human.
- **AC2 (soft boundary still works at the new size):** Given the expanded bounds, when the player drives to any edge of the new playable area, then the existing soft-boundary behavior (a gentle clamp keeping the truck within bounds, per `drive-terrain-and-gas.md` AC4) continues to work exactly as it does today, just at the larger extent — no void, no getting stuck at the edge, no behavior change other than the edge being farther away.
- **AC3 (existing content remains functional in place):** Given the expanded bounds, the three existing obstacles (bush, rock, derelict car), the four existing structures (windmill, barn, farmhouse, mountain), and the river remain at their current coordinates, fully reachable by driving, and every existing behavior tied to them (wheel-tier clearance per obstacle, solid-structure blocking, spawn keep-out zones, river's zero mechanical effect) continues to work unchanged. Nothing about these features regresses just because the surrounding empty area got larger.
- **AC4 (spawns use the full new area):** Given the expanded bounds, the animal, farmer, and fuel-pickup spawn systems select valid random locations across the *entire* new 100×100 area (not just within the old 40×40 footprint), while continuing to respect all existing keep-out zones around obstacles/structures.

### Hills (visual only)

- **AC5 (hills visible across the expanded terrain):** Given the expanded terrain, when the player drives around the drivable area, then the ground visibly shows rolling, rounded, gentle elevation changes ("hills") rather than reading as a flat plane — consistent with a golf-course-like character (smooth, rounded, moderate in height), not canyons, cliffs, or jagged terrain.
- **AC6 (hills are visually distinct from the mountain landmark):** Hills read at a clearly smaller scale than the existing single mountain landmark (`environment-dressing.md` AC3a) — the mountain remains the map's one dramatic large feature; hills are a background rolling-terrain texture, not a second set of mountains.
- **AC7 (driving over a hill produces a visible climb response):** Given the truck driving across a hill, when its position crosses the hill's rendered footprint, then the truck's rendered body visibly rises and tilts to follow the hill's contour, in the same visual language as the existing obstacle-climb system (`truck-obstacle-climbing.md` AC1) — the hill must read as real terrain underneath the truck, not a flat-shaded backdrop the truck ignores.
- **AC8 (hard safety constraint — truck movement is unaffected by hills, testable):** Given any hill anywhere on the expanded terrain, when the truck drives over, along, or across it at any angle or speed, then: (a) the truck's actual position, speed, and steering math are computed exactly as they are on flat ground — no slope-based deceleration, drift, traction change, or forced steering; (b) the truck never becomes stuck (unable to proceed in the direction of input) because of a hill; (c) the truck never flips, spins, launches, or is thrown into any orientation a flat-ground drive couldn't also produce; (d) no player input precision or timing is required to safely cross any hill. This must hold for every hill on the map and at every approach angle, not just a hand-picked case — this is the doc's core safety constraint and must be demonstrably testable (e.g., automated coverage that asserts a truck's simulated position/velocity are identical whether or not hill data is present, given the same inputs).
- **AC9 (hills are not a clearance-gated obstacle class):** Hills never enter the wheel-tier clearance system (`core/clearance.ts`) as either `blocking` or `passable` — driving over a hill is available identically to a truck of any wheel tier, unlike bush/rock/derelict car.
- **AC10 (hills are perceivable without special camera work):** Given the default driving camera, hills are visible from ordinary positions within the drivable area (not requiring a specific vantage point, freecam, or map-edge view) — a player driving around in a normal session actually sees the rolling terrain, not just a flat-looking scene that happens to have elevation data underneath.

Absent a stated preference, this doc only requires the qualitative "gentle, rolling, golf-course-like, visually distinct from the one mountain landmark" character described in AC5/AC6.

## Constraints

- **Must not change the truck's actual movement/physics model.** The truck's driving math (`core/driving/`) stays pure 2D and kinematic, per `docs/architecture/0001-foundation-stack-and-structure.md` §2 — hills are a rendering-layer addition only, using the same "position-derived, stateless, no real vertical physics axis" pattern already established and confirmed by the human for the existing obstacle-climb system (`truck-obstacle-climbing.md` Open Question 1's resolution).
- **Must not regress the existing obstacle-climb visual system.** The four-corner sampling that already handles bush/rock/derelict-car climbing (`docs/architecture/0014-obstacle-climb-four-corner-sampling.md`) must continue to produce correct results for those existing obstacles once terrain height is added into the same calculation — this doc extends that system, it does not replace or fork it.
- **Must not alter the wheel-tier clearance rule, spawn keep-out logic, or the coordinates of existing obstacles/structures/river** — those are governed by `drive-terrain-and-gas.md` AC5-AC9, `environment-dressing.md` AC6, and `core/terrain.ts`'s existing data respectively; this doc only widens the world around them (see AC3, Open Question 1).
- Runs in-browser (Three.js + Vite) — any new hill geometry/rendering approach falls under the same in-browser performance expectations already established for the game's other terrain/environment content (`vehicle-and-character-art.md`'s shared perf/loading section); this doc does not set a new, separate budget, but a materially larger terrain mesh is a real perf consideration the architect should size, not assume free.
- The exact new `TERRAIN_BOUNDS` values (e.g., -50..50 vs. some other ~100×100-equivalent split) are an implementation detail for the architect — the requirement is the ~6x area increase (AC1), not a specific numeric boundary.
