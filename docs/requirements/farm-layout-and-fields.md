# Corn/Wheat Fields, Realistic Farm Layout, and New Farmstead Structures

Status: Draft — requirements only, **implementation must not start before the sibling map-expansion epic ships** (see Constraints). Not yet attached to a sprint. Fence collidability (originally Open Question 1) resolved 2026-07-10 — see AC8.

Related: `docs/requirements/environment-dressing.md` (direct precedent for `StructureInstance`, spawn keep-out, art-direction rules, and the "scope split into independently trackable issues" convention this doc reuses — see "Scope split" below); `docs/architecture/0012-environment-dressing-and-terrain-features.md` (design this doc's structures half should extend, not replace); `docs/requirements/drive-terrain-and-gas.md` (`TERRAIN_BOUNDS`, existing wheel-tier clearance system this doc must not disturb); `docs/requirements/truck-builder-and-upgrades.md` / `docs/architecture/0002-upgrade-tier-data-model.md` (the wheel-tier clearance table, again — protected here too); the sibling **map-expansion requirements doc** (being written in parallel, not yet landed as of this writing) — this epic consumes that epic's final `TERRAIN_BOUNDS`, it does not define them.

## Problem statement

Two direct player asks plus one already-approved bonus are bundled here because they all touch "how the farm world is composed": (1) the farm has no crops — it's grass and a handful of standalone obstacles/structures, which doesn't read as a working farm; (2) the five existing structures (windmill, barn, farmhouse, mountain, plus the three stub obstacles) were placed ad hoc within the current small 40×40 bounds, mostly to keep them apart from each other and clear of the truck's start — not to read as a real farm layout, and the player has directly asked for this to look intentional; (3) the team already has unused, already-licensed fence/silo/chicken-coop models sitting in the same Quaternius pack family used for the barn/windmill (issue #46), so adding them is a near-zero sourcing-cost way to round out a proper farmyard rather than leaving the barn/farmhouse standing alone.

## Goals / Non-goals

**Goals**
- Add corn and wheat fields that are visually recognizable and drivable-through with zero mechanical effect on the truck, matching the already-locked-in "textured/colored ground patch + ~15-30 low-poly stalk-cluster props scattered near the edges" scope (confirmed by the human this session — **not** dense per-stalk rendering; see Non-goals).
- Re-place the existing structures (windmill, barn, farmhouse, mountain) so the overall arrangement reads as a coherent, intentional farm layout once the map is larger, instead of today's "spread apart so nothing overlaps" placement.
- Add silo and chicken-coop structures, reusing already-licensed, unused assets from the same Quaternius Farm Buildings Bundle already partially used for the barn/windmill (issue #46) — near-zero incremental sourcing cost.
- Add fences that read as marking a field/pen boundary as part of that same coherent layout.
- Resolve, per new structure, whether it's a physical obstacle or pure backdrop, so the developer/architect has an unambiguous rule (same discipline `environment-dressing.md` AC1-AC9 established for the first five structures) — with the fence case flagged explicitly as unresolved (see Open Questions).
- Keep the existing wheel-tier clearance system (bush/rock/derelict-car, `drive-terrain-and-gas.md` AC5-AC9) completely untouched — this is additive/rearranging dressing, not a rebalance. Same invariant every environment-dressing feature so far has protected.

**Non-goals**
- **Dense, per-stalk-instanced field rendering.** Already decided by the human this session: this codebase has zero `THREE.InstancedMesh` precedent, and full-density individually-rendered stalks was assessed as the single highest-engineering-risk item on the roadmap for comparatively modest extra delight over patch+sparse-props. Out of scope for this doc; a future doc could revisit if there's appetite once instancing has been proven out elsewhere.
- **No new gameplay mechanic tied to fields** — no crushing, no per-stalk scoring, no "hide from the farmer in the corn" mechanic, no friction/slowdown while driving through a field. This mirrors the exact precedent `environment-dressing.md` set for the river ("fording/bridging... is a real gameplay feature, not a dressing task... risks scope creep into physics/movement rules that belong in a dedicated requirements pass if the human wants the river to matter mechanically later"). The locked patch+sparse-props scope also makes a stalk-level mechanic (e.g. hiding) awkward to build convincingly anyway, since most of a field is a flat textured patch, not individually-modeled stalks. Judged not to have a strong enough case to escalate as an open question — if the human wants a fields mechanic later, that's cleanly a separate future story, not a blocker here.
- **The actual map-size change itself** (40×40 → ~100×100) is out of scope for this doc — that belongs to the sibling map-expansion epic. This epic only consumes the new `TERRAIN_BOUNDS` once that epic ships; see Constraints.
- **No sound design** — consistent with `CLAUDE.md`'s project-wide sound deferral and `environment-dressing.md`'s identical non-goal.
- **No animation requirement** for field stalks or fence geometry beyond whatever's cheap/free from the chosen assets — static is acceptable, matching the existing windmill/river precedent (`environment-dressing.md` AC8).
- **Exact literal coordinates are not specified by this doc.** As with the first five structures, "realistic layout" is a content/design judgment call evaluated by looking at the running scene (screenshots + iteration), not by locking coordinates in a requirements doc. The acceptance criteria below describe the *qualities* of a good layout, not a level-design spec.

## Scope split

Following the exact precedent and rationale set by `environment-dressing.md`'s own "Scope split" section (its structures-vs-terrain-features split, approved by the human 2026-07-09): this doc's three bundled asks split naturally into **two** issues along an implementation-dependency line, not three, because two of the three asks are tightly coupled to each other while the third is architecturally independent:

- **Farmstead layout & new structures** — re-placing the four existing structures (windmill, barn, farmhouse, mountain) *and* adding the new silo, chicken-coop, and fence structures, as one issue. Rationale for bundling these two asks together (unlike the environment-dressing split, which kept unrelated things apart): a "realistic layout" re-pass and "where do the new buildings go" are the *same* placement decision — you cannot sensibly decide where the silo and chicken coop belong without simultaneously deciding the barn/farmhouse cluster they belong next to, and re-laying out the existing four structures without also seating the new ones would mean redoing the same layout work twice. All of it uses the identical `StructureInstance`/`createStructureColliders`/spawn-keep-out pipeline already built for issue #46/#47, so there's no rendering-approach split to justify separate tracking either.
- **Corn/wheat fields** — visually and architecturally independent of the structures work: different asset shape (ground-patch + sparse props, closer to the river's procedural/decorative precedent than to the buildings' collidable-structure precedent), no shared implementation dependency, and either can ship first without blocking the other. Matches the environment-dressing precedent's own reasoning almost exactly (structures vs. a decorative terrain-style feature).

Both issues share the same dependency on the sibling map-expansion epic (see Constraints) and the same cross-cutting rules below (wheel-tier clearance untouched, spawn keep-out extended, asset-load-never-crashes, art-direction consistency) — this is a scope/tracking split, not a design fork, exactly as `environment-dressing.md` states for its own split.

## User stories

1. As a player, I want corn fields and wheat fields on the farm that I can drive through, so the world has more visual variety than flat green ground. *(fields)*
2. As a player, I want the farm's buildings and fences arranged like a real farmyard — not scattered randomly — so the world feels like an actual place instead of a set of disconnected props. *(layout & new structures)*
3. As a player, I want to see a silo and a chicken coop as part of the farmstead, so the farm reads as more complete/lived-in. *(layout & new structures)*

## Acceptance criteria

### Fields — corn, wheat

- **AC1 (fields visually recognizable):** Given the drivable farm scene, when a player looks at a field, then it is recognizable at a glance as either "corn" or "wheat" via distinct ground texture/color plus the scattered stalk-cluster props near its edges — mirrors `environment-dressing.md` AC9's silhouette-recognizable-at-a-glance bar, applied to a ground-patch feature instead of a structure.
- **AC2 (fields are purely decorative, zero mechanical effect):** Given a corn or wheat field placed on the terrain, when the truck drives across, along, or through it (including through or near individual stalk-cluster props), then truck movement, top speed, gas drain/regen, and coin systems are completely unaffected — zero mechanical effect, directly mirroring `environment-dressing.md` AC4's river pattern ("the river has zero mechanical effect this sprint"). Stalk-cluster props are non-collidable decorative geometry, not `ObstacleInstance`s or `StructureInstance`s with colliders.
- **AC3 (patch + sparse props, not dense instancing):** Given a field, when it is rendered, then it consists of a textured/colored ground patch plus roughly 15-30 individual low-poly stalk-cluster props scattered near its edges — not a dense field of individually-rendered/collidable stalks. This is the locked-in scope decision (see Non-goals); an implementation that renders substantially more individual stalk geometry than this, or gives stalks colliders, does not satisfy this AC.
- **AC4 (fields positioned plausibly relative to the farmstead):** Given the re-laid-out farmstead (structures issue) and the fields, when a player looks at the overall arrangement, then the fields read as belonging to the farm — positioned near/adjacent to the farmstead cluster in a way a child would recognize as "these are the farm's fields," not scattered independently across the map. Evaluated by looking at the running scene, not by coordinate assertions (see Non-goals).

### Farmstead layout & new structures — windmill/barn/farmhouse/mountain re-placement, silo, chicken coop, fences

- **AC5 (existing structures genuinely re-placed, not left as-is):** Given the sibling map-expansion epic's new `TERRAIN_BOUNDS` have landed, when this epic is implemented, then the four existing structures (windmill, barn, farmhouse, mountain) are moved to new positions appropriate to the larger bounds and the new farmstead cluster described in AC6 — "leave them at their current 40×40-derived coordinates" does not satisfy this AC even if those coordinates happen to still fall inside the new bounds.
- **AC6 (coherent farmstead grouping):** Given the full set of structures (windmill, barn, farmhouse, mountain, silo, chicken coop, fences) placed within the new bounds, when a player looks at the farmstead area, then the structures read as belonging to one sensible functional cluster — e.g. barn/farmhouse/silo/coop grouped near each other as a farmyard, fences marking a field/pen boundary near that cluster, the windmill and mountain standing as visible landmarks rather than being placed for mere spacing — rather than looking randomly scattered. Evaluated via in-game screenshots and iteration, per this project's established layout-review convention (`environment-dressing.md` open questions #1/#2 precedent), not by literal coordinates locked in this doc.
- **AC7 (silo and chicken coop are solid, collidable, no-fail-state, like existing buildings):** Given the silo and chicken coop placed within the drivable bounds, when the truck attempts to drive into either, then the truck is blocked (cannot pass through) regardless of wheel tier, with no damage, no hit against body capacity, and no fail-state penalty — identical behavior to `environment-dressing.md` AC2 for windmill/barn/farmhouse, implemented via the same `StructureInstance`/`collidable: true` pattern.
- **AC8 (fence is a breakable barrier — resolved 2026-07-10, supersedes the original binary framing):** The human resolved Open Question 1 with a third option neither of the doc's original two choices covered: fences start as a **real, solid barrier** (blocks the truck like AC7's silo/coop — solid, no-fail-state, tier-independent) but **collapse/break open on truck contact**, after which that fence segment no longer blocks the truck for the remainder of the driving session. This is a deliberate, positive "smash through the fence" interaction consistent with the monster-truck fantasy and this project's forgiving, no-fail-state design bias — not a damage/penalty mechanic (the truck takes no hit, no coin/score consequence either way; this is closer in spirit to the farmer's TIRED/LEAVING state transition — a stateful, one-way visual change triggered by contact — than to the fuel-pickup collected/not-collected toggle, though architecturally it may resemble either). Concretely:
  - **Standing state:** solid collider, blocks the truck exactly like AC7's silo/coop, until contact.
  - **Contact → collapse:** when the truck makes contact with a still-standing fence segment, that segment transitions to a "collapsed" visual (e.g. knocked-over/broken-open pose, asset-dependent) and its collider is removed/shrunk so the truck can then pass through where it stood.
  - **Reset behavior:** fences reset to their standing state at the start of each new driving session (consistent with how every other per-session dynamic element in this game — animals, farmer, fuel pickups — resets on a fresh session rather than persisting state across restarts; this project has no save-game persistence beyond local storage per `CLAUDE.md`'s stated out-of-scope list).
  - **Non-blocking tuning details left to the architect/developer** (does not need further human sign-off): whether any contact triggers collapse or a minimum approach speed/force is required (recommend: any contact, for simplicity and the game's established forgiving bias), and the exact visual collapse treatment (asset-dependent — a "broken fence" pose if the sourced Quaternius asset ships one, or a simple tip-over/fade otherwise, matching AC10's "graceful fallback" spirit).
  - **Architectural note for the next pipeline stage:** unlike every other `StructureInstance` shipped so far (windmill/barn/farmhouse/mountain/silo/coop — all static for the life of a session), a fence segment now needs *mutable* per-session state (standing vs. collapsed). This is new territory for `StructureInstance` and should be scoped explicitly by the architect — e.g. whether fences get their own sibling type with a `broken: boolean` (or similar) field alongside the existing static structures, contact-detected the same 2D-distance way `core/boop.ts`/farmer-contact checks already work (not via reading raw Rapier collision events, to stay consistent with this project's established `core/`-purity contact-check pattern) — rather than overloading the existing `collidable: boolean` flag, which was designed for a static true/false, not a runtime transition.
- **AC9 (spawn-avoidance extended to new solid structures):** Given the silo, chicken coop, and *standing* (not-yet-collapsed) fences occupy space with solid colliders, when the animal spawn system (`animal-chase-and-coins.md` AC1), the farmer spawn system (`farmer-minimal-bump.md` AC1), or the fuel-pickup spawn system (`fuel-pickups.md`) pick a random valid location, then none of them spawn an entity inside/overlapping a new solid structure's collider — extends the existing "not inside an obstacle" rule exactly as `environment-dressing.md` AC6 did for the first three buildings. (Non-blocking implementation note: since fences can collapse mid-session, whether spawn keep-out re-evaluates a collapsed fence's footprint as open is a minor judgment call — recommend leaving collapsed-fence footprints in the keep-out set regardless, i.e. treat keep-out as based on the fence's original standing footprint for simplicity, since spawn positions are picked once at spawn-time, not continuously re-validated.)
- **AC10 (asset load never crashes the game):** Given any of the new structures' art assets (silo, chicken coop, fence) fails to load, when the scene initializes, then the game does not crash or hang — it falls back to a simple placeholder and logs a warning, identical to `environment-dressing.md` AC7.
- **AC11 (art direction and pack-family consistency):** Silo, chicken coop, and fence assets are sourced from the same Quaternius Farm Buildings Bundle pack family already used for the barn and windmill (issue #46), in the confirmed stylized/low-poly-but-recognizable art direction (`vehicle-and-character-art.md` "Resolved — Art direction") — silhouette-recognizable as "silo"/"chicken coop"/"fence" at a glance, consistent in style with the rest of the world, matching `environment-dressing.md` AC9.

### Cross-cutting — applies to both halves

- **AC12 (existing wheel-tier clearance system unchanged):** Neither the fields nor the new/re-placed structures introduce a new entry in the wheel-tier clearance table (`ObstacleClass`) or change the existing bush/rock/derelict-car behavior specified in `drive-terrain-and-gas.md` AC5-AC9. Those acceptance criteria continue to pass unmodified — identical invariant to `environment-dressing.md` AC5, protected the same way (fields and new structures must never enter `partitionObstacles`/`clearance.ts`).
- **AC13 (this epic consumes bounds, it does not define them):** This epic's layout and field placement are implemented against whatever `TERRAIN_BOUNDS` the sibling map-expansion epic ships (target ~100×100 per that epic's own scope) — this doc does not itself change `TERRAIN_BOUNDS`, and implementation must not begin until that epic has landed (see Constraints).

## Constraints

- **Hard dependency: this epic's implementation must not start before the sibling map-expansion epic ships.** The map needs to actually be bigger (~100×100, up from today's 40×40) before a "realistic layout" re-placement of windmill/barn/farmhouse/mountain plus new content (fields, fences, silo, chicken coop) makes sense — today's structures are hand-placed within the small 40×40 bounds specifically because there wasn't much room (see `core/terrain.ts`'s own placement comments). Requirements can be (and are, by this doc) written in parallel with that epic, but design/build should wait for its final `TERRAIN_BOUNDS` to land.
- Must not alter or re-tune the existing wheel-tier clearance system (`drive-terrain-and-gas.md` AC5-AC9, `truck-builder-and-upgrades.md` wheel table) — see AC12. Applies to both split issues.
- Silo, chicken coop, and fence assets should be sourced by reusing the already-licensed, unused Quaternius Farm Buildings Bundle assets already partially consumed for the barn/windmill (issue #46) — near-zero incremental sourcing cost is the whole rationale for this being an "approved bonus" rather than a separately-scoped art-sourcing epic; a genuinely new sourcing pass should only happen if that bundle turns out not to have suitable silo/coop/fence models on inspection.
- Runs in-browser (Three.js + Vite); glTF is the established asset format (`docs/architecture/0001-foundation-stack-and-structure.md`, `docs/architecture/0010-art-asset-pipeline-and-loading.md`). Perf/loading budget is the same shared budget established in `vehicle-and-character-art.md`'s Non-functional section — this doc's new assets (silo, coop, fence, field ground textures, stalk-cluster props) count against that shared budget, not a separate one.
- No `THREE.InstancedMesh` precedent exists in this codebase today (verified) — the patch+sparse-props field design (AC3) is deliberately chosen to avoid needing to establish one for this feature; if dense instanced fields are wanted later, that's new scope requiring its own design pass, not an assumed extension of this doc.
- Target player is a young child: fields and new structures should read clearly as what they are at a glance (silhouette/texture-recognizable), not abstract/ambiguous — matches the confirmed stylized/low-poly art direction (`vehicle-and-character-art.md`) and the same bar `environment-dressing.md` set for the first five structures.
