# Truck Body Lift — Authentic Monster-Truck Stance

Status: Draft — no decisions locked in by the human yet (unlike `truck-scale-and-suspension.md`'s three pre-confirmed decisions); every open question below is genuinely open and should be resolved (by the architect proposing defaults, confirmed by human playtest, or by the human directly) before this is sized and scheduled into a sprint.

Related: `docs/requirements/truck-scale-and-suspension.md` (issue #62/#63 — the size-up and independent per-wheel suspension this doc's baseline-gap change directly interacts with, per the issue's own framing); `docs/architecture/0018-truck-scale-and-independent-wheel-suspension.md` (ADR 0018 — §1's `TRUCK_SCALE`/socket-data-folding approach and §3's `SuspensionConfig`/`maxTravel` are the mechanisms this doc's lift is measured against); `docs/requirements/truck-obstacle-climbing.md` (the existing whole-body lift/tilt effect that must keep working at the new baseline height); `docs/requirements/truck-wheel-motion.md` (existing wheel roll/steer-yaw that must keep working at the new baseline height); `docs/requirements/truck-builder-and-upgrades.md` (the 3 body tiers and their existing size progression this doc must not flatten); `docs/requirements/vehicle-and-character-art.md` (the tier-to-tier visual-progression design intent, same precedent `truck-scale-and-suspension.md` AC2 cited); `src/render/truck-sockets.ts` (`BODY_TIER_SOCKETS`'s per-tier body/wheel Y-offset data — the documented convention today is "the body's underside rests at wheel-center height," which is the specific thing this doc asks to change); `src/render/truck-rig.ts` (`buildTruckRig()` — the single rig-assembly path shared by the driving scene and the builder preview, ADR 0011 §4).

## Problem statement

Direct human feedback while reviewing the recently-shipped #62 (bigger truck) and #63 (independent per-wheel suspension): the truck body currently sits too low/close to the wheels. `truck-sockets.ts`'s own documented convention places each tier's body so its underside rests at wheel-*center* height (not above the wheel), which reads as a low, squat stance rather than the tall, lifted stance real monster trucks have. This also has a concrete, already-shipped downstream consequence: #63's per-wheel suspension travel (`DEFAULT_SUSPENSION_CONFIG.maxTravel`, currently 0.25 world units) has very little vertical room between body and wheel to actually be seen moving in — the effect works correctly today but is hard to notice, because the baseline gap it's offsetting from is small.

## Goals / Non-goals

**Goals**
- Increase the vertical gap between the truck body's underside and the wheel axle/top-of-wheel line, for all 3 body tiers, so the truck reads as a lifted "monster truck" stance at a glance (not just bigger overall, per #62, but visibly *taller relative to its own wheels*).
- Preserve the existing, already-meaningful tier-to-tier size/height progression (Tier 0 < Tier 1 < Tier 2) — the lift must not flatten or erase the relative size difference between tiers established by `truck-builder-and-upgrades.md` and reinforced by #62's proportional scale-up.
- Make the already-shipped per-wheel suspension travel (#63) clearly legible against the new baseline gap — a wheel's up/down travel should read as visible motion, neither imperceptible (too little headroom, the problem today) nor so exaggerated relative to the new gap that a wheel appears to detach/float away from the body or punch through it at maximum travel.
- Keep the existing whole-body obstacle-climb lift/tilt (`truck-obstacle-climbing.md`) and wheel roll/steer-yaw (`truck-wheel-motion.md`) working correctly at the new, taller baseline — both are unchanged mechanisms, just measured from a different starting height.
- Keep the truck reading as a stable, grounded vehicle at the new stance — consistent with the project's forgiving, non-chaotic design bias for a young player (ADR 0001 §1).

**Non-goals**
- Any change to `TRUCK_SCALE` or the truck's overall length/width/silhouette size — that is #62's already-shipped concern. This doc is a vertical body-*position* change only, not another size-up pass.
- Any change to `TRUCK_CONTACT_RADIUS` or the truck's horizontal collision/contact radius (the reach used for animal boop, farmer bump, fuel pickup, and obstacle-climb footprint sampling) — those are horizontal-plane values under the kinematic, XZ-planar physics model (ADR 0001 §2); raising the body vertically has no obvious reason to touch them, though this is confirmed rather than assumed (see Open Question 5).
- Any change to the functional value of the four upgrade axes — hit capacity (body), obstacle-clearance class (wheels), top speed (engine), gas capacity (gas tank) stay numerically identical to today's values for every tier. This is a visual-stance change, not a rebalance, same "no rebalance" line `truck-scale-and-suspension.md` AC5 already drew.
- Any change to the wheel-roll or front-wheel steer-yaw *mechanism* (`truck-wheel-motion.md`) or the whole-body climb lift/tilt *mechanism* (`truck-obstacle-climbing.md`) — both keep working exactly as designed today; only the static baseline height they're layered on top of changes.
- Any change to which obstacles are `blocking`/`passable` or the wheel-tier/obstacle-class clearance rule (`core/clearance.ts`'s `canClear()`) — untouched by this doc.
- Re-authoring the sourced body/wheel glTF models' own geometry (e.g., moving a fender's modeled wheel-well cutout) — in scope here is the rig's socket Y-offset (a `truck-sockets.ts` data change), not editing the source assets themselves. If a live-render pass finds the existing fender geometry doesn't have enough headroom to look right at a raised gap (Open Question 3), that finding is surfaced back through this doc's process, not silently worked around by re-authoring assets.
- Whether and how far the already-shipped suspension's amplitude constants (`travelGain`/`maxTravel`) get re-tuned as a consequence of the new baseline — that re-tuning is explicitly in scope to *check* (AC3) but the resulting numbers are a downstream tuning decision for the architect/developer + playtest, not fixed by this doc, same treatment `truck-scale-and-suspension.md`'s Open Question 3 gave the original suspension amplitude.

## User stories

1. As a player, I want my truck's body to sit noticeably higher above its wheels, so it looks like a real, tall monster truck instead of a low, squashed-down vehicle.
2. As a player, I want the size difference between my body upgrade tiers to stay clearly noticeable after the lift, so upgrading still feels meaningful.
3. As a player, I want to actually see each wheel's independent up-and-down suspension motion when I drive over a bump, so the suspension effect I already have feels worth having.
4. As a player, I want my taller-standing truck to still look stable and properly attached to its wheels (not floating, tipping, or gapped-open underneath), so it doesn't look broken.

## Acceptance criteria

- **AC1 (bigger body-to-wheel gap, all tiers):** Given any of the 3 body tiers, when its rendered stance is compared to today's shipped stance for that same tier, then the vertical gap between the body's underside and the wheel axle/top-of-wheel line is measurably and noticeably larger — large enough that a player recognizes "the body sits higher / there's daylight under the truck now" at a glance. Exact magnitude is a tuning value, not fixed by this AC (Open Question 1).
- **AC2 (tier-to-tier progression preserved):** Given the three body tiers rendered side by side after the lift, when a player compares them, then Tier 0 < Tier 1 < Tier 2 remains clearly and distinguishably ordered by size/height, at a relative gap comparable to today's shipped progression — the lift must not flatten the tiers toward a uniform stance or make two adjacent tiers hard to tell apart. Whether the lift amount itself varies by tier or is applied uniformly is Open Question 2; either approach must satisfy this AC.
- **AC3 (suspension travel legible against the new baseline):** Given the truck driving over a `passable` obstacle with one wheel corner engaged (the same scenario as `truck-scale-and-suspension.md` AC6), when that wheel's existing per-wheel suspension travel (#63) is observed, then the vertical motion reads as clearly visible against the new baseline gap — not imperceptible (today's defect) and not so large relative to the new gap that the wheel appears to visually detach/float away from the body's underside or clip through it at maximum travel. Verified by human playtest/screenshot against the pre-change baseline, same discipline as ADR 0018's own climb re-tune (not code-inspection-only).
- **AC4 (existing whole-body climb and wheel motion keep working):** Given the same obstacle-crossing and steering scenarios already covered by `truck-obstacle-climbing.md` AC1 and `truck-wheel-motion.md` AC1-AC6, then the existing whole-body lift/tilt response and the existing wheel roll/steer-yaw animation continue to occur exactly as they do today, correctly, at the new baseline height — neither mechanism needs to change to keep working, and neither is visually broken by the taller stance.
- **AC5 (no clipping or disconnect artifacts):** Given the truck at rest or in motion at the new stance, then the body's underside does not visually clip through the tops of the wheels, and there is no unintended visible gap/hole between the body and the wheel-well area that breaks the "one attached vehicle" read (e.g., seeing straight through where a fender should visually meet its wheel) — each wheel stays visually anchored under the body's fender/wheel-well silhouette, just with more clearance above it.
- **AC6 (functional stats and horizontal hitbox unchanged, all tiers/axes):** Given a truck built at any body/wheel/engine/gas-tank tier, then that tier's hit capacity, obstacle-clearance class, top speed, gas capacity, and horizontal collision/contact radius (`TRUCK_CONTACT_RADIUS`) are numerically identical to their values before this feature — this is a vertical visual repositioning only, not a rebalance and not a hitbox-size change (also see Open Question 5 on the physics collider's own vertical extent).
- **AC7 (no chaotic or unstable read):** The taller stance does not make the truck look like it will tip, wobble, or float, and requires no precision timing/input from the player to look correct — consistent with the young-child-friendly, forgiving design bias already established project-wide (ADR 0001 §1), same standing constraint `truck-obstacle-climbing.md` AC3 and `truck-scale-and-suspension.md` AC10 already carry.
- **AC8 (builder preview reflects the new stance):** Given the builder's stationary 3D truck preview (`buildTruckRig()`, ADR 0011 §4 — the same single rig-assembly path the driving scene uses), then the preview shows the new lifted stance identically to the driving scene, with no preview-specific divergence — same precedent `truck-scale-and-suspension.md`'s Non-goals section already established for this preview (it may omit suspension *motion*, since it has no driving state, but must still reflect the new static geometry).

## Open questions

1. **(Non-blocking, tuning) Exact lift magnitude.** How much taller is "monster truck tall" for this game's art style and young-player audience? Left to the architect/developer to propose a default and the human to confirm via playtest — any increase that satisfies AC1's "reads as taller/lifted at a glance" bar is acceptable at requirements level. Same treatment as `truck-scale-and-suspension.md` Open Question 2's size-magnitude tuning value.
2. **(Needs an architect proposal, not just tuning) Does the lift amount vary by body tier, or is it a single shared value applied identically to all 3 tiers?** This is the issue's own explicitly flagged question. A single global/proportional lift (analogous to ADR 0018 §1's single global `TRUCK_SCALE` factor) is the simplest option and worth considering as a default proposal, but the architect should confirm it satisfies AC2 (tier progression preserved) rather than assuming it — a bigger tier's already-bigger wheels may need a proportionally bigger gap to read consistently, or a flat gap may look fine; this needs a live render comparison, not a guess.
3. **(Needs a live-render check, not decidable from requirements) Does the existing body-model wheel-well/fender geometry have enough headroom above the wheel silhouette for a raised gap to look right without re-authoring the sourced assets?** Per this doc's Non-goals, re-authoring the source glTF geometry is out of scope — but if a live screenshot pass finds the fender's modeled cutout doesn't have room for the chosen lift magnitude (e.g., the wheel starts poking out the side/top of its modeled wheel-well opening), that's a real constraint the architect needs to surface back to this doc's process before implementation proceeds, not something to silently work around.
4. **(Non-blocking, tuning) Does `DEFAULT_SUSPENSION_CONFIG.maxTravel` (currently 0.25 world units) and/or `DEFAULT_CLIMB_CONFIG`'s lift/tilt constants need re-tuning to look proportionate against the new, taller baseline gap?** AC3 requires this to be *checked*; the resulting numbers (if any change is needed) are a downstream playtest-driven tuning decision, not fixed here — same treatment `truck-scale-and-suspension.md` gave its own original suspension-amplitude open question.
5. **(Needs explicit confirmation, not an assumption) Does the physics collider's vertical extent (`TRUCK_HALF_HEIGHT`, `src/systems/driving-system.ts`) need any change?** This doc's presumption (stated in Non-goals) is "no" — the truck's physics stay kinematic and XZ-planar (ADR 0001 §2), and this is purely a render-layer socket Y-offset change on the visual rig, not a physics-collider change. The human's own issue report explicitly asked "does it affect the collision hitbox / center-of-mass" — the architect should confirm this presumption in writing rather than leave it implicit, given that exact question was raised directly.
6. **(Informational, not a hard requirement of this doc) Interaction with two already-open camera bugs.** Issue #66 (chase camera clips into structure geometry) and issue #67 (builder preview crops Tier 1/2 bodies at the #62 size) are pre-existing, already-filed bugs unrelated to this doc's own scope. A visually taller truck could plausibly interact with or exacerbate either (more silhouette to clip, more height to crop) — flagged for the architect's awareness when sequencing/sizing this doc's implementation, not as a requirement this doc must fix.

## Constraints

- Must not change `TRUCK_SCALE`, `TRUCK_CONTACT_RADIUS`, or any of the four upgrade axes' functional values for any tier (AC6) — this is a visual-stance change only, not a rebalance or another size-up pass.
- Must not change the existing wheel-tier/obstacle-class clearance *rule* (`core/clearance.ts`'s `canClear()`/`partitionObstacles()`) or which obstacles are `blocking`/`passable`.
- Must not fork the single-truck-rig-assembly invariant (ADR 0011 §4 — the driving scene and the builder preview both call the same `buildTruckRig()`) — the builder preview must reflect the new static stance (AC8).
- Whatever mechanism is chosen must stay within the existing arcade-kinematic architecture (ADR 0001 §2) — this is a `truck-sockets.ts`/`truck-rig.ts` render-layer socket-offset change, never a real physics/vertical axis on the Rapier collider or the kinematic controller.
- Must not regress ADR 0001's target-player design bias (forgiving, predictable, never chaotic) — a hard constraint given the young-child audience, same standing constraint carried by every sibling truck-visual doc (`truck-obstacle-climbing.md`, `truck-scale-and-suspension.md`).
- Likely lands in the same seam ADR 0018 already extended — `src/render/truck-sockets.ts`'s `BODY_TIER_SOCKETS` per-tier body Y-offset data and `src/render/truck-rig.ts`'s `buildTruckRig()` — whether this is best recorded as a new ADR or a dated addendum to ADR 0018 (per this project's "supersede via strikethrough + dated note" convention) is the architect's call, not decided here.
- Runs in-browser (Three.js + Vite + Rapier3D-compat), consistent with the rest of the render/physics layers — no new tech introduced by this doc.
